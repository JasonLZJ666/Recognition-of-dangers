## 一、完整答辩演讲稿

### 第 1 页：封面

各位老师好，我汇报的题目是《危险标志识别系统》。

本项目的目标是针对附件中给出的危险警示标志，完成一个从数据处理、模型训练到网页端演示的完整识别系统。系统最终可以识别 5 类危险标志，分别是爆炸风险、高处坠物、落石危险、易燃危险和水域危险。

在实现方式上，我们没有只做一个静态页面，而是把模型训练、数据增强、后端推理、前端交互和部署说明都做成了可运行的完整流程。接下来我会从项目目标、系统架构、模型训练、前端实现和测试结果几个方面进行汇报。

### 第 2 页：项目一句话说明

这一页是整个项目的核心概括：我们要把附件中的 5 类危险标志，训练成一个可以在网页中实时识别的完整系统。

这里有三个关键词。第一个是“5 类危险标志”，说明我们的识别目标是明确的，不是泛泛地识别所有交通标志，而是围绕课程附件中的危险场景进行分类。第二个是“不同角度和光线”，因为实际拍摄时标志可能会有旋转、透视、亮度变化和背景干扰。第三个是“完整系统”，也就是不仅有模型文件，还要有可以演示、可以复现、可以部署的前端和后端。

因此本项目的技术路线不是单一模型训练，而是围绕最终可用性来设计：前期整理数据，中间训练模型，后期把训练好的模型接入网页端，并提供启动脚本和部署文档。

### 第 3 页：问题与挑战

危险标志识别看起来是一个简单的图像分类任务，但真正做成可演示系统时，会遇到几个问题。

第一，原始素材数量有限。如果只用附件里的少量图片训练，模型很容易记住图片本身，而不是学习危险标志的视觉特征。这样在前端换一个角度或换一张照片时，就容易识别错误。

第二，真实拍摄环境不稳定。摄像头画面可能存在倾斜、过曝、阴影、模糊和背景干扰。尤其是黄色三角形警示标志，如果背景里也有类似颜色，模型前处理阶段就可能裁剪错误。

第三，浏览器端直接加载 ONNX 模型存在兼容性问题。之前我们尝试过 ONNX Runtime Web，但在不同浏览器和本地环境中可能出现 WASM 后端加载失败的问题。为了保证答辩演示稳定，当前版本改为 Python 后端自动加载 PyTorch 模型，前端负责交互展示。

所以这个项目需要解决的不只是“训练一个分类器”，还包括数据增强、稳定推理、前后端联动和部署复现。

### 第 4 页：系统架构

本系统采用浏览器加 Python 后端的结构。

浏览器端主要负责三件事：上传图片、调用摄像头、展示识别结果。用户可以选择图片上传，也可以直接拖拽图片，还可以打开摄像头进行单帧识别或实时识别。

Python 后端负责核心推理流程。前端把图片以 base64 的形式发送给后端，后端接收后会进行图像解码、方向修正、危险标志区域定位、裁剪、归一化，然后调用 PyTorch 中加载的 EfficientNet-B0 模型进行预测。最后后端把预测类别、置信度、风险等级、处置建议和推理耗时返回给网页。

这样的设计有两个好处。第一，模型运行环境更稳定，因为 PyTorch 在 Python 中直接加载 `.pt` 权重，不依赖浏览器的 ONNX Web 后端。第二，前端逻辑更清晰，网页只负责用户交互和结果展示，模型推理由后端统一完成。

### 第 5 页：数据集构建

数据集部分是本项目中非常重要的一环。

原始数据来自附件中的危险标志素材，我们先整理出 5 个类别，并统一类别名称和目录结构。这 5 类分别对应爆炸风险、高处坠物、落石危险、易燃危险和水域危险。

考虑到原始样本较少，项目进一步构建了 `dataset_viewpoint` 视角和光照增强数据集。增强方式包括旋转、缩放、透视变换、亮度变化、对比度变化、模糊、阴影和背景扰动。这些增强不是为了单纯增加图片数量，而是为了模拟答辩现场或实际使用中可能出现的拍摄条件。

比如同一个危险标志，在手机摄像头前可能会倾斜，也可能因为灯光导致颜色变浅，还可能因为距离变化导致图像变小。通过增强数据训练，模型更容易学习标志的稳定特征，而不是只记住某一张原图。

### 第 6 页：模型选择

模型部分当前推荐使用 EfficientNet-B0。

EfficientNet-B0 的特点是参数规模不大，但分类能力比较强。它通过复合缩放的思想，在网络深度、宽度和输入分辨率之间取得平衡。对于本项目这种类别数不多、样本来自课程素材并经过增强的小型图像分类任务，EfficientNet-B0 在准确率、模型体积和推理速度之间比较合适。

训练时我们使用迁移学习。也就是说，模型主干部分利用已有的图像特征提取能力，然后替换最后的分类头，让它输出本项目的 5 个危险标志类别。当前配置中冻结了大部分主干参数，主要训练分类头，这样可以降低小样本训练时的过拟合风险，也能减少训练时间。

除了 EfficientNet-B0，训练代码中还保留了 ResNet18、MobileNetV3-Small 和自定义 StrongCNN 等结构，方便做对比实验和展示扩展性。但最终答辩演示时，我们使用表现更稳定的 EfficientNet-B0 作为默认模型。

### 第 7 页：训练流程

训练流程主要由 `model/train_model.py` 作为入口，其他模块负责数据读取、模型构建、指标计算、训练循环和报告导出。

训练时，输入是增强后的 `dataset_viewpoint` 数据集，输出目录是 `model/artifacts_viewpoint`。训练完成后会保存最佳模型 `best_danger_sign_model.pt`，同时保存训练配置、类别顺序和相关评估文件。

优化策略上，我们使用 AdamW 优化器，并配合学习率调度、Label Smoothing、Dropout 和早停机制。AdamW 可以在训练中提供更稳定的权重更新，Label Smoothing 能降低模型对训练样本的过度自信，Dropout 用于缓解过拟合，早停机制则用于在验证指标不再提升时停止训练。

评价指标不只看 Accuracy，还包括 Top-1、Top-3、Macro-F1 和逐类别指标。这样做的原因是，如果只看总体准确率，可能会掩盖某个类别表现较差的问题；而 Macro-F1 会更关注每个类别的平均表现。

### 第 8 页：图像预处理

前端识别准确率不仅取决于模型本身，也取决于送入模型的图像是否合理。因此我们在 Python 后端加入了图像预处理步骤。

用户上传图片或摄像头截图后，后端会先读取图像并进行方向修正，然后尝试定位危险标志区域。由于危险标志通常具有明显的黄色区域和黑色图形，系统会结合颜色和暗色像素信息寻找主要连通区域，并裁剪出更接近标志本体的图像。

裁剪完成后，图像会被缩放到模型需要的输入尺寸，并按照训练时相同的均值和方差进行归一化。这样可以保证前端实际输入和训练阶段的数据分布尽量一致。

这一部分非常关键。之前模型识别效果不稳定时，一个重要原因就是前端输入和训练输入不一致，或者裁剪区域受背景影响。后续我们通过改进裁剪逻辑和使用更贴近拍摄场景的数据集，提高了前端图片识别和摄像头识别的稳定性。

### 第 9 页：Python 前端

这一页展示的是当前推荐使用的 Python 前端版本。

系统启动时，会自动加载 `model/artifacts_viewpoint/best_danger_sign_model.pt`。用户只需要运行启动脚本，然后在浏览器打开 `http://127.0.0.1:7860/`，就可以进入识别页面。

前端支持四种主要操作：图片上传识别、拖拽图片识别、摄像头单帧识别和摄像头实时识别。识别结果会显示类别名称、置信度、风险等级、处置建议、Top-5 候选结果和推理耗时。

相比纯静态网页，这个版本更适合答辩和验收。因为它能明确展示“模型已经被导入系统”，并且每次识别都会经过后端真实推理，而不是前端写死结果。

### 第 10 页：测试结果

从测试结果来看，当前模型在本项目构建的数据集上表现较好。

在验证集上，模型 Top-1 准确率和 Macro-F1 都达到了较高水平。独立增强测试集 `dataset_test_viewpoint` 上，Top-1 为 0.9975，Macro-F1 为 0.9975。前端测试照片集 `test_inputs/frontend_photos` 中，30 张测试图片全部识别正确，结果为 30/30。

这里需要说明的是，这些指标反映的是在当前课程数据、增强数据和前端测试照片范围内的表现，并不代表在所有真实世界场景中都能达到完全相同的准确率。后续如果要投入真实应用，还需要采集更多现场图片继续扩充数据集。

但是对于课程设计和本地演示来说，这些结果说明当前训练流程、预处理方式和前后端推理链路是有效的。

### 第 12 页：部署与复现

运行这个系统主要需要 Python、PyTorch、Torchvision、Pillow 和 NumPy。复制项目时，核心文件包括 `python_frontend`、`model`、`dataset`、`app`、`docs` 以及已经训练好的模型文件。

部署文档中还说明了如何检查模型是否加载成功，如何访问 `/api/status`，以及如何通过 `--checkpoint` 参数指定其他训练好的模型。

因此这个项目不是只能在我本机运行，而是已经整理成了可迁移、可复现的形式。

### 第 13 页：工程问题与解决

项目过程中遇到的一个主要问题是 ONNX 浏览器端加载不稳定。

最开始我们尝试把模型导出成 ONNX，然后直接在浏览器中通过 ONNX Runtime Web 运行。但在本地启动时出现过 WASM 后端模块加载失败的问题，例如浏览器找不到相关的 `.mjs` 或 `.wasm` 文件。这类问题和浏览器环境、资源路径、本地服务器方式都有关系，排查成本较高。

为了解决演示稳定性问题，我们调整为 Python 后端推理。这样启动系统时直接加载 PyTorch 的 `.pt` 模型，不再依赖浏览器端的 ONNX Runtime Web。前端仍然是网页形式，但推理过程由 Python 后端完成。

另一个问题是模型识别结果一度偏向某个类别。后来我们从两个方面改进：一是重建视角和光照增强数据集，让训练样本更接近真实拍摄条件；二是改进前端裁剪逻辑，让模型看到的图像更接近标志本体，而不是被背景干扰。

这些问题的处理也体现了本项目不是只完成代码编写，而是经过了调试、验证和方案迭代。

本项目的工作量主要体现在四条线。

第二条是模型线。我们实现了 EfficientNet-B0 迁移学习训练流程，并保留多模型结构用于扩展。训练代码中包含数据增强、优化器、学习率调度、早停、指标统计和模型保存。

第三条是系统线。我们实现了 Python Web 前端，支持图片上传、拖拽识别、摄像头单帧识别和实时识别，并且启动时自动加载训练好的模型。

第四条是文档线。项目包含 README、技术报告、部署到其他电脑的运行说明、答辩 PPT 和本演讲稿。这样从代码到演示材料都比较完整，方便课程验收。

### 第 15 页：不足与后续优化

虽然当前系统已经可以完成演示，但仍然有改进空间。

第一，当前数据主要来自附件和程序增强，真实现场照片还不够多。后续可以采集更多不同距离、不同背景、不同设备拍摄的图片，提高模型对真实环境的泛化能力。

第二，目前系统的主要任务是分类，也就是判断画面中的标志属于哪一类。如果未来需要处理更复杂场景，比如一张图里有多个危险标志，可以进一步扩展为目标检测模型，例如 YOLO 系列。

第三，部署方式还可以进一步工程化。现在需要安装 Python 和依赖，后续可以封装成可执行文件，或者使用 Docker 镜像，让其他电脑运行更方便。

第四，摄像头实时识别可以继续优化帧率和稳定性，比如加入连续帧投票机制，减少单帧误判。

在系统方面，我们实现了 Python 前端推理版本，启动时自动加载训练好的模型，支持图片上传、拖拽识别、摄像头单帧识别和实时识别。

在工程交付方面，我们整理了 README、技术报告、部署说明和答辩材料，使项目能够在本地稳定演示，也能够迁移到其他电脑运行。

## 三、常见问题回答准备

### 1. 为什么选择 EfficientNet-B0？

因为 EfficientNet-B0 参数量相对较小，分类能力较强，适合本项目这种 5 类小规模图像分类任务。相比更大的模型，它训练和推理成本更低；相比简单 CNN，它利用 ImageNet 预训练特征，泛化能力更好。

### 2. 为什么要做数据增强？

原始附件素材数量有限，如果直接训练容易过拟合。实际拍摄中会有角度、光照、距离和背景变化，所以我们通过旋转、缩放、透视、亮度、对比度、阴影和背景扰动来模拟真实输入，提高模型鲁棒性。

### 3. 为什么不用浏览器直接加载 ONNX？

我们尝试过 ONNX Web 方案，但浏览器端依赖 WASM 后端和本地资源路径，容易出现兼容性问题。为了保证答辩和验收演示稳定，当前改用 Python 后端直接加载 PyTorch 模型，前端只负责交互展示。

### 4. 当前准确率能说明什么？

当前准确率说明在本项目构建的数据集、独立增强测试集和前端测试照片范围内，模型效果较好。但它不能等同于真实世界所有场景的准确率。如果后续用于真实现场，还需要继续采集现场图片并重新训练和评估。

### 5. 如果以后要提高系统能力，可以怎么做？

可以从三个方向改进：第一，采集更多真实场景数据；第二，把分类任务扩展成目标检测任务，支持一张图中多个标志；第三，把部署方式封装为可执行文件或 Docker 镜像，让使用更加方便。

## 四、3 分钟压缩版

各位老师好，我汇报的题目是《危险标志识别系统》。

本项目面向附件中给出的危险警示标志，完成了一个从数据处理、模型训练到网页端演示的完整识别系统。系统可以识别 5 类危险标志，包括爆炸风险、高处坠物、落石危险、易燃危险和水域危险。

项目的主要难点有三个。第一，原始素材数量有限，直接训练容易过拟合。第二，实际拍摄时会出现角度、光照和背景变化。第三，浏览器端直接加载 ONNX 模型容易遇到 WASM 后端兼容问题。针对这些问题，我们构建了视角和光照增强数据集，并采用 Python 后端自动加载 PyTorch 模型的方式完成推理。

在模型方面，我们使用 EfficientNet-B0 迁移学习。它在模型体积、推理速度和分类性能之间比较均衡，适合本项目这种小规模 5 分类任务。训练时冻结主干特征提取部分，主要训练分类头，并使用 AdamW、Label Smoothing、Dropout、学习率调度和早停机制来提高训练稳定性。

在系统方面，浏览器负责图片上传、拖拽图片和摄像头采集，Python 后端负责图像裁剪、归一化和模型推理。启动系统时会自动加载 `best_danger_sign_model.pt`，识别结果会返回类别、置信度、风险等级和处置建议。

测试结果方面，当前模型在独立增强测试集上的 Top-1 为 0.9975，Macro-F1 为 0.9975；在前端测试照片集上为 30/30。这个结果说明当前训练流程、预处理方式和前后端推理链路在课程数据范围内是有效的。
