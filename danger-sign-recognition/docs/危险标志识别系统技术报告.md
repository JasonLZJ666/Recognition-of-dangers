# 危险标志识别系统技术报告

## 摘要

本项目面向危险警示标志自动识别场景，设计并实现了一套集数据处理、模型训练、图像预处理、Python 后端推理和网页端交互于一体的危险标志识别系统。系统支持对图片和摄像头画面中的危险标志进行识别，能够输出预测类别、置信度、风险等级和处置建议。

项目当前采用 Python 前端版本作为主要交付方案。该方案在服务启动阶段自动加载已经训练完成的 PyTorch 模型，网页端仅负责采集图片或摄像头画面，实际识别过程由 Python 后端完成。相比浏览器端 ONNX Runtime Web 方案，Python 后端推理避免了浏览器 WASM 后端兼容性问题，部署和演示稳定性更高。

模型部分采用 EfficientNet-B0 迁移学习结构，以 5 类危险标志为识别目标，通过视角、光照、透视、阴影和背景扰动等数据增强方式提升模型对真实拍摄条件的适应能力。当前推荐模型在验证集上的 Top-1 准确率为 0.9946，Macro-F1 为 0.9946；在独立增强测试集上的 Top-1 为 0.9975；在前端测试照片集上的识别结果为 30/30。

**关键词：** 危险标志识别；EfficientNet-B0；PyTorch；图像分类；Python Web 前端；迁移学习；数据增强

## 1. 项目背景

危险警示标志广泛出现在道路、施工现场、实验场所、仓储区域和公共安全管理场景中。传统人工识别方式依赖人员观察，存在效率低、漏检风险高、实时性不足等问题。随着深度学习和计算机视觉技术的发展，可以利用图像分类模型对危险标志进行自动识别，为安全巡检、风险提示和智能监控提供辅助。

本项目围绕课程设计和系统验收需求，完成以下工作：

- 整理危险标志基础素材并建立标准数据集。
- 构建视角和光照增强数据集，提高模型泛化能力。
- 设计 EfficientNet-B0 迁移学习训练方案。
- 实现模型训练、评估、报告导出和模型保存流程。
- 实现 Python 版本网页前端，启动时自动加载训练好的模型。
- 支持图片上传、拖拽识别、摄像头单帧识别和实时识别。

## 3. 系统总体架构

当前推荐版本采用 B/S 架构，但推理任务放在 Python 后端执行。整体结构如下：

```text
浏览器页面
  ├─ 图片上传
  ├─ 拖拽图片
  ├─ 摄像头采集
  └─ 结果可视化
        ↓ HTTP JSON
Python Web 后端
  ├─ 接收 base64 图片
  ├─ 图像解码与 EXIF 修正
  ├─ 危险标志区域定位与裁剪
  ├─ 图像归一化
  ├─ PyTorch 模型推理
  └─ 返回类别、置信度、裁剪框、耗时
        ↓
EfficientNet-B0 分类模型
```

系统核心文件如下：

```text
python_frontend/server.py                  Python Web 服务和推理逻辑
python_frontend/start_python_frontend.ps1  一键启动脚本
model/artifacts_viewpoint/best_danger_sign_model.pt  当前推荐模型
model/architectures.py                     模型结构定义
model/train_model.py                       训练入口
model/evaluate_model.py                    测试集评估
model/evaluate_frontend_photos.py          前端裁剪一致性评估
```

该架构的优势是推理环境受控，避免浏览器 WebAssembly、ONNX Runtime Web、跨域文件读取等兼容性问题。对于课程验收和现场演示，Python 后端方案更稳定，也更容易解释“模型已自动导入”这一过程。

## 4. 数据集建设

### 4.1 基础数据

项目最初从附件/PDF 素材中提取 5 类危险标志，并按类别整理为：

```text
dataset/
  explosion/
  falling_objects/
  falling_rocks/
  flammable/
  water_hazard/
```

基础数据集用于确定类别体系和标准标志样式。

### 4.2 视角与光照增强数据集

为了让模型适应实际拍摄场景，项目构建了 `dataset_viewpoint/` 数据集。该数据集在保留 5 类原始危险标志类别的基础上，通过程序生成不同视角、光照和背景变化的样本。

增强方式包括：

- 随机旋转
- 尺度缩放
- 仿射变换
- 透视倾斜
- 亮度变化
- 对比度变化
- 轻微模糊
- 阴影扰动
- 背景扰动

根据训练配置，`dataset_viewpoint/` 共包含 5 类、1105 张图片。该数据集不引入新的类别，目的是让模型仍然识别附件中的 5 类危险标志，同时具备对拍摄角度和光线变化的适应能力。

### 4.3 独立测试集

项目还构建了 `dataset_test_viewpoint/` 作为独立测试集。该测试集使用不同随机种子生成，用于验证模型是否真正具备泛化能力，而不是只记住训练样本。

## 5. 模型设计

### 5.1 模型选型

本项目支持多种模型结构：

```text
efficientnet_b0
resnet18
mobilenet_v3_small
strong_cnn
```

当前推荐使用 EfficientNet-B0。EfficientNet 通过复合缩放策略同时平衡网络深度、宽度和输入分辨率，在较小参数规模下具有较好的分类性能。对于本项目这种小规模、多增强样本的图像分类任务，EfficientNet-B0 在准确率、模型体积和推理速度之间比较均衡。

### 5.2 迁移学习策略

训练配置如下：

| 参数 | 配置 |
|---|---|
| 模型结构 | EfficientNet-B0 |
| 是否使用预训练 | 是 |
| 是否冻结 backbone | 是 |
| 输入尺寸 | 224 x 224 |
| 类别数 | 5 |
| 训练轮数 | 30 |
| Batch Size | 16 |
| 学习率 | 0.0002 |
| Weight Decay | 0.0001 |
| Dropout | 0.35 |
| Label Smoothing | 0.05 |
| 最优指标 | `val_macro_f1` |
| 优化器 | AdamW |

迁移学习时冻结 EfficientNet-B0 的特征提取部分，只训练分类头。这样做可以减少小样本训练时的过拟合风险，同时显著降低训练参数量。

模型参数统计：

| 指标 | 数值 |
|---|---:|
| 总参数量 | 4,336,769 |
| 可训练参数量 | 329,221 |
| 冻结参数量 | 4,007,548 |
| 可训练参数比例 | 7.59% |
| 估计模型大小 | 16.704 MB |

### 5.3 分类头设计

模型保留 EfficientNet-B0 的主干特征提取能力，并替换末端分类头。分类头包含 Dropout、Linear、SiLU 激活和最终分类层。Dropout 用于降低过拟合，SiLU 激活用于提升非线性表达能力，最终输出 5 个类别的 logits。

## 7. 图像预处理与裁剪

实际图片或摄像头画面通常包含大量背景，例如墙面、地面、光斑、阴影和白色边框。如果直接将整张图送入分类模型，背景会干扰判断，导致模型偏向某些类别。因此系统在推理前加入危险标志区域定位与裁剪步骤。

预处理流程如下：

```text
输入图片
  ↓
EXIF 方向修正
  ↓
转换为 RGB
  ↓
扫描黄色警示主体
  ↓
结合黑色边框/图案区域确认标志位置
  ↓
扩展为带留白的方形裁剪框
  ↓
Resize 到 224 x 224
  ↓
ImageNet mean/std 归一化
  ↓
送入 PyTorch 模型
```

该裁剪逻辑位于：

```text
model/evaluate_frontend_photos.py
python_frontend/server.py
```

裁剪算法不是简单地寻找最大黄色区域，而是要求黄色主体附近存在黑色边框或图案支撑，避免背景中的黄色光斑、装饰色块被误认为警示标志。这一点对摄像头实时识别非常重要。

## 8. Python 前端设计

Python 前端由 `python_frontend/server.py` 实现，使用标准库 `http.server` 提供网页和接口服务，不额外引入 Flask 等框架，部署简单。

### 8.1 启动流程

启动脚本：

```powershell
powershell -ExecutionPolicy Bypass -File python_frontend\start_python_frontend.ps1
```

启动时执行以下操作：

1. 清理旧的同端口服务。
2. 选择默认 checkpoint：

   ```text
   model\artifacts_viewpoint\best_danger_sign_model.pt
   ```

3. 调用 `server.py`。
4. 在 Python 进程中加载 PyTorch 模型。
5. 在浏览器中访问 `http://127.0.0.1:7860/`。

### 8.2 后端接口

| 接口 | 方法 | 功能 |
|---|---|---|
| `/` | GET | 返回网页前端 |
| `/api/status` | GET | 返回当前模型加载状态 |
| `/api/predict` | POST | 接收 base64 图片并返回识别结果 |

`/api/status` 返回模型结构、类别数、设备、checkpoint 路径等信息。`/api/predict` 返回最优类别、5 类置信度、推理耗时、裁剪框和模型信息。

### 8.3 前端交互

网页端支持：

- 图片文件选择
- 图片拖拽上传
- 摄像头开启/关闭
- 摄像头单帧识别
- 摄像头实时识别
- 结果置信度条形图展示
- 风险等级和处置建议展示

网页端不直接运行深度学习模型，因此对浏览器环境要求较低。只要浏览器可以访问本地服务并授权摄像头，即可完成识别。

## 11. 与旧版 ONNX 前端对比

项目早期使用 ONNX Runtime Web 在浏览器中直接加载 `danger_sign_model.onnx` 和 `.onnx.data` 文件进行推理。该方案的优点是推理完全在浏览器侧完成，但也存在以下问题：

- 浏览器必须正确加载 ONNX Runtime Web 的 WASM 后端。
- 部分环境会出现 `.jsep.mjs` 动态模块加载失败。
- 用户不能直接双击 HTML 文件，需要通过本地 HTTP 服务访问。
- 浏览器缓存可能导致旧脚本仍被执行。

Python 前端版本将模型推理放回后端，启动时直接加载 `.pt` 模型，规避了 ONNX Web 兼容性问题。对于本地演示、答辩和验收，Python 版本更稳定，也更容易展示模型自动导入过程。

## 12. 不足与改进方向

当前系统已经实现了完整的训练、推理和网页展示流程，但仍有以下可改进点：

1. **真实数据规模仍需扩大。** 当前数据主要来自附件素材和程序增强，后续可以采集真实场景照片，补充更多光照、遮挡和拍摄距离变化。
2. **检测能力仍以单标志分类为主。** 当前系统先定位显著警示主体再分类，若画面中同时存在多个危险标志，仍需要进一步引入目标检测模型。
3. **摄像头实时识别可继续优化。** 后续可以加入多帧投票、置信度阈值、低置信度人工复核提示，减少画面抖动导致的瞬时误判。
4. **部署方式可进一步工程化。** 可以将 Python 前端封装为可执行文件或 Docker 镜像，降低环境配置难度。
5. **模型可以继续蒸馏或量化。** 如果需要在低性能设备上运行，可以尝试 MobileNetV3、模型量化或 TorchScript 导出。
